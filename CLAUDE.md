# CLAUDE.md - Net Worth Tracker (Lean)

## Project Overview
Net Worth Tracker is a Next.js app for Italian investors to track net worth, assets, cashflow, dividends, performance metrics, and long-term planning with Firebase.

## Current Status
- Stack: Next.js 16, React 19, TypeScript 5, Tailwind v4, Firebase, Vitest, Framer Motion, Recharts, Yahoo Finance, Borsa Italiana scraping, Anthropic (Claude Sonnet 5)
- Latest (2026-07-14): **Allocazione — `AllocationRole` (tradable / frozen / excluded) + "Preleva" (decumulation)**. The page assumed every euro of net worth is tradable in fractions. It is not: you cannot sell €3.500 of an apartment, and a pension fund is locked until retirement. Both used to inflate the DENOMINATOR (so every other class's `targetValue = target% × totalValue` was measured against untouchable wealth), peg their class permanently off-target (Balance Score stuck low), and make Ribilancia order an impossible `VENDI Immobili −185.000 €`.
  - **(1) `Asset.allocationRole`** (`types/assets.ts`; 3-way select in `AssetDialog`, any asset type). It first shipped as a single `excludeFromAllocation` boolean and was reworked in-session: that boolean conflated **two orthogonal questions** — *"is this part of my invested portfolio?"* × *"can I trade it?"*. Hence one field, three values: **`tradable`** (default — in the denominator, in the plans); **`frozen`** (pension fund, private equity — IS invested wealth, so it stays **in the denominator and the percentages** but never appears in a plan; the plans then reach the target by moving what you CAN move, and *that automatic compensation is the whole point* — dropping a bond-heavy pension fund out of the totals would show a 60/40 free portfolio while the TRUE exposure is far more conservative); **`excluded`** (the home you live in — not an investment, so **out of the page entirely, denominator included**; keeping it in pegs realestate off-target forever against a `VENDI` nobody can execute).
  - `partitionByAllocationRole` runs **upstream of `compareAllocations`** (which receives `tradable + frozen`) — filtering downstream would measure every other `targetValue` against the wrong base AND break the Σ(current − target) = 0 invariant `computeBalanceScore` halves. Legacy `excludeFromAllocation: true` reads as **`excluded`, never `frozen`** (behaviour-preserving; upgrading it would move the user's numbers unasked). `frozen` and `excluded` get **separate, clickable** hero captions because they have opposite relations to the headline — frozen is INSIDE it, excluded is OUTSIDE it (merging them into one "non ribilanciabili" figure is the easy, wrong thing); plus an "Esclusi dall'allocazione" group in `AllocationBreakdown` and a note under the Settings target editor.
  - **Executability**: `splitFromSurplus` gained a **`capacity`** input distinct from `currentValue` — the surplus is measured on the FULL value (a frozen fund really does push its class above target; pretending otherwise understates the drift) but the take is capped at the TRADABLE slice. `buildRebalancePlan` caps the SELL side at `tradableByClass` (never the BUY side), keeping `requestedAmount` + `limitedByFrozen` so the panel prints a fillable order while the `% → %` line still shows the true drift.
  - The trap the roles set — a target stranded *because it covered the house* — is caught by `findOrphanedTargets` + an inline fix-it, and bites hardest **at the sub-category level**: a `realestate: { Prima casa 70%, REIT 30% }` config emptied "Prima casa" but kept its 70% target, so Versa poured money into a bucket that can only hold the excluded house (found in real use; fixed by also `stripOrphanedSubTargets`-ing the map handed to the planners and the breakdown — warning alone is not enough). A class is NOT orphaned while any sub-target stays reachable (you can buy a REIT holding none), and an *unfunded* target is not an orphan — only excluded value behind it makes one.
  - **(2) "Preleva"**, third mode of `ActionPlanner`: `splitFromSurplus` is the mirror of `splitTowardTarget`, draining what sits ABOVE target first so a withdrawal *rebalances* instead of distorting. Invariant asserted throughout: **`Σamount === min(requested, Σcapacity)`** at every level. Versa and Preleva are now **ONE `PlanNode` tree with the sign flipped**, shared `PlanRow`, both drilling **class → sub-category → instrument**; levels with no target fall back to exact pro-rata for free via synthetic "neutral targets" through the same split. Their **asymmetry is the design**: *you can be told to buy what you do not own, never to sell it* — Versa's buckets come from the configured targets and honour the specific-asset targets ("compra EPRA" while you hold none), Preleva's come strictly from what is HELD. Capital-gains tax is out of scope (plan is per class, `taxRate` is per asset).

## Known Issues (Active)
- FX conversion depends on Frankfurter availability; 24h cache fallback. Pre-migration non-EUR assets without `currentPriceEur` show native price as EUR until the first price update (GBp safe via the pence→GBP `/100` fallback).
- Demo account requires manual Firebase setup (user + realistic fake Firestore data + three env vars).
- Security review 2026-06-10: all 8 findings (SEC-1…SEC-8) done (`docs/security-review-spec.md`). Caveat: firebase-admin pinned at `^13.6.0` (the @14 bump pulled pure-ESM `jose@6` → `ERR_REQUIRE_ESM` on Vercel), so the 8 moderate `uuid` advisories stay open until resolvable.
- **Shared account setup**: (1) the guest's email must be in `REGISTRATION_WHITELIST` and they must **register their own account before** the owner can add them (member add resolves email→uid via Admin — a non-existent user 404s). (2) `firestore.rules` must be deployed (`firebase deploy --only firestore:rules`) for the client-SDK shared path to work — the app code only enables the feature once the app itself is deployed. (3) Theme/`userPreferences` intentionally follows the viewer, not the shared account.
- YOC/Current-Yield are keyed by `assetId` and exclude sold assets (intended). **Discontinuous-holding edge — mitigated 2026-06-13**: a sold-then-rebought instrument reuses the old `assetId` (ISIN continuity in `createAsset`, or `quantity 0`→>0), so a window spanning the gap used to credit the prior holding's dividends against the new cost. Now `yieldOnCost.ts` drops dividends paid before `Asset.holdingStartDate` (stamped at (re)purchase in `createAsset`/`updateAsset`; snapshot fallback `deriveHoldingStartDates`); the same scoping is applied to Rendimento Totale per Asset (`stats/route.ts`). DPS-growth still counts all received dividends by design. Caveat: the snapshot fallback is monthly-granular and needs a recorded gap; legacy rebuys lack the stamp until re-added.
- **TWR cash-flow / liquidity month-bucket artifact (by design, not a formula bug)**: monthly snapshots freeze end-of-month NW; TWR neutralizes a tracked expense only if the NW drop and the expense's cash flow fall in the SAME month. A late-month event whose balance is updated the following month (e.g. taxes/credit-card spending paid on the next statement) splits into an overstated leg and a phantom-negative leg that don't fully cancel. It's data-entry timing, not code — record balance changes in the month of competence, or model the credit card as a liability so the NW drop lands at expense time. Cash/liquidity IS part of the return base (no exclusion). Full explanation in AGENTS.md (History and Snapshot Baselines → TWR neutralization).

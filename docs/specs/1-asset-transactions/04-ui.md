# 04 — UI Surfaces (v1)

Read **DESIGN.md** before any of this. All user-facing copy Italian, code comments English.
Sign colors ONLY via theme tokens `text-positive` / `text-destructive` (never raw green/red —
AGENTS.md *Layout Tokens*). Watch for the curly-apostrophe TS1127 trap in Italian strings.
Forms: `useWatch()` for render-time reads, `getValues()` in handlers, never `watch()`.
Every mutating control: `disabled={isDemo}` + `aria-label` demo pattern. Every dialog:
description required. Buttons that are not submits: explicit `type="button"`.

IA principle: **Patrimonio = positions and operations; Rendimenti = returns.** Nothing about
trades appears on Panoramica in v1.

## 1. Entry points (Patrimonio — `app/dashboard/assets/page.tsx` + `components/assets/AssetManagementTab.tsx`)

- Per-row actions for ledger-type assets (desktop table AND mobile `AssetCard`):
  - **"Registra operazione"** → opens `TransactionDialog` with the asset preselected.
  - **"Movimenti"** → opens `AssetMovementsDialog`.
  - Keep icon buttons ≥ `h-8 w-8` with `aria-label` (touch-target rule). Non-ledger assets
    (cash, realestate) show neither action.
- Both actions are hidden until `useAssetLedgerMeta` resolves to "migrated" (spec 03 §4 trigger
  runs on this page; while meta is absent the page is exactly today's page).
- Remember the dynamic `colSpan` gotcha if a column is added to the table — but v1 adds row
  ACTIONS, not columns; the G/P, PMC and Δ columns are untouched.

## 2. `components/assets/TransactionDialog.tsx` (new)

`ResponsiveModal` (the convergence target for form modals) + react-hook-form + zod. Props:
`{ open, onClose, asset, transaction? }` — `transaction` present = edit mode (same
field-visibility logic, the AssetDialog edit-reuse precedent).

- **Type selector**: segmented control **Compra / Vendi / Rettifica** (DESIGN.md segmented pill,
  Framer `layoutId`, spring 400/35 — the `ActionPlanner` precedent). Disabled in edit mode
  (changing a trade's type is a delete+recreate, keep v1 simple) and locked for baselines.
- **Fields (Compra / Vendi)**:
  - `Data` — default today; min = `meta.baselineDate`, max = today. Helper text when clamped:
    `"Le operazioni partono dal {baselineDate} (inizio del registro)."`
  - `Quantità` — number > 0; step free (crypto fractions). For Vendi: current held quantity shown
    as helper + **"Vendi tutto"** inline button setting the exact quantity.
  - `Prezzo per unità` — native currency, suffix shows `asset.currency`. For bonds with
    `bondDetails`: accept the Borsa Italiana %-of-par quote and convert with the SAME
    `resolveBondPrice` helper AssetDialog uses (label: `"Prezzo (quotazione Borsa Italiana)"`).
    For GBp listings the input is GBP (already normalized) — never pence.
  - `Commissioni (€)` — optional, ≥ 0.
  - `Conto di regolamento` — Radix Select over the user's cash assets (`assetClass === 'cash'`),
    first option `"Nessuno"` with sentinel `__none__` (NEVER empty string). Helper:
    `"Se selezionato, il saldo del conto viene aggiornato automaticamente."`
  - `Note` — optional.
- **Fields (Rettifica)**: `Data`, `Nuova quantità` (≥ 0), `Nuovo PMC` (per unit, native), `Note`.
  One explainer line: `"Imposta la posizione da questa data: per split, correzioni o strumenti
  valorizzati a versamento. Nessuna plusvalenza realizzata."` No fees, no settlement account.
- **Live summary** (below the fields, updates via `useWatch`):
  - Compra: `"Totale: {qty×price + fees} €"`.
  - Vendi: total + **estimated realized P&L preview** — trades are client-readable, so run the
    SAME pure replay on `[...existingTrades, draftSell]` and show
    `"P&L realizzato stimato: ±X €"` with sign tokens. One shared engine = preview always agrees
    with the server (Cross-Component Metric Consistency rule). Note: `priceEur` is server-resolved
    (spec 01 §6), so for non-EUR assets the client builds the draft's `priceEur` with the asset's
    current conversion ratio (`currentPriceEur / currentPrice`) — hence "stimato" in the label;
    the authoritative figure comes back in the mutation response (spec 03 §2).
- **Submit**: mutation hook → on error show the server's Italian message (422 bodies carry
  `userMessage`); success toast AFTER the request resolves (never before — the
  toast-after-reconcile rule). Dialog-reset effect must include `open` in deps + `if (!open)
  return` + full new-record field enumeration (Dialog Form Reset Pattern).

## 3. `components/assets/AssetDialog.tsx` changes

- **Edit mode, ledger asset types**: the `Quantità` and Cost-Basis (`averageCost`) inputs are
  replaced by a read-only summary block:
  `"Quantità e PMC sono gestiti dal registro operazioni."` + current values + a button
  `"Registra operazione"` that closes AssetDialog and opens TransactionDialog. The
  "Tracciamento Cost Basis" toggle and the "Calcola PMC" calculator disappear for these types in
  edit mode. The two advisory hints ("Hai investito nuovo capitale? / Hai venduto…") are removed
  for ledger types (the ledger replaced them). **Submit path switches to `updateAssetMetadata`**
  (spec 03 §3) — sending the old payload through `updateAsset` would `deleteField()` the PMC.
- **Create mode, ledger asset types**: keep quantity/price fields but reframe as the opening
  trade: section header `"Posizione iniziale (primo acquisto)"`, fields `Quantità`,
  `Prezzo di acquisto per unità` (feeds both `averageCost` and the first buy), `Data di acquisto`
  (default today, min `baselineDate`), optional `Conto di regolamento`. Flow: `createAsset` with
  quantity 0 and no averageCost → then `POST /api/asset-transactions` for the first buy (which
  writes the derived quantity/PMC onto the asset). If step 2 fails, keep the dialog open with the
  error and the created asset visible at quantity 0 — recoverable state, and a Why-comment
  documents the accepted non-atomicity of the two steps.
- Cash and realestate flows: completely unchanged.

## 4. `components/assets/AssetMovementsDialog.tsx` (new)

`ResponsiveModal`, opened per asset. Query `useAssetTransactions(ownerId, asset.id)` gated on
`isOpen` (lazy-load rule).

- **Header strip** (the asset's ledger vitals, from the pure engine + live value):
  `P&L realizzato` (sign tokens) · `Rendimento totale` (`computeAssetTotalReturn`, includes
  dividends; Popover explaining the composition) · `XIRR` (`"–"` when null, Popover:
  `"Rendimento annualizzato ponderato per i flussi (XIRR), dalle date reali delle operazioni."`).
- **List** (date desc, day-grouped not needed — flat `divide-y` rows per DESIGN.md list idiom):
  each row = date (`formatDate`) · type chip (`Compra` positive tint / `Vendi` destructive tint /
  `Rettifica` neutral / baseline: outline chip `Posizione iniziale`) · `qty × price` (mono) ·
  total EUR · fees when present · for sells the realized P&L of THAT sale (computed by replay —
  expose per-transaction realized in the engine or recompute prefix-replay in a small helper) ·
  actions Edit / Delete.
- **Delete**: 2-click inline confirm with 3s auto-disarm (the `AssetManagementTab` precedent),
  then DELETE route; remind the user in the confirm label when the trade had a settlement:
  `"Conferma? Il saldo del conto verrà ristornato."`
- **Baseline row**: no delete button; edit opens TransactionDialog limited to
  quantity/PMC/note (spec 03 §1 baseline protection).
- Empty state (ledger asset with no trades — post-migration zero-quantity assets):
  `"Nessuna operazione registrata. Registra il primo acquisto per aprire la posizione."`

## 5. Rendimenti (`app/dashboard/performance/page.tsx`) — Phase D

- **"Capitale investito"**: a `MetricCard` row inside the existing Contributi/metric sections
  showing `netInvestedEur` from `computeInvestedCapital` over the SAME selected period as the
  page (respect the period selector; pass period bounds down, don't recompute a different window
  — Cross-Component Metric Consistency). Its info Popover MUST disambiguate the two neighbors:
  `"Capitale investito: acquisti meno vendite registrati nel registro operazioni nel periodo.
  Contributi netti: risparmio esterno stimato dal tracciamento spese. Misurano cose diverse."`
  (the Risparmio-Netto-vs-Rapporto disambiguation precedent — two numbers that will never match
  need their relationship stated on screen.)
- **"Plusvalenze realizzate"**: new `MetricSection` inside the existing "Mostra tutte le metriche"
  Collapsible — per-year rows (year · realized EUR with sign tokens) from the aggregated
  `realizedByYear` across all assets, plus a total row. Empty state: section hidden (progressive
  disclosure — no empty shells).
- Data source: client-side — `useAssetTransactions(ownerId)` (all trades) + pure helpers. No new
  endpoint needed in v1. Both blocks are gated on `useAssetLedgerMeta` resolving to "migrated"
  (a user whose first visit is Rendimenti simply doesn't see them until Patrimonio has triggered
  the migration — same degrade-to-today behavior as the Patrimonio entry points).

## 6. `totalReturnAssets` extension (`app/api/dividends/stats/route.ts`) — Phase D

- For assets WITH ledger entries (Admin-read trades grouped by assetId): compute via
  `replayTransactions` + `computeAssetTotalReturn` — includes closed positions (add them to the
  response with `isClosed: true`) and partial sells; dividend term = the already-scoped net
  dividends the route computes today. Response items gain optional fields
  `{ realizedPnlEur?, isClosed? }` — additive, so existing consumers keep working (the
  cache-schema-evolution rule: optional fields, no breaking shape change).
- Assets without any ledger doc (only possible for positions opened and never migrated — i.e.
  quantity-0-at-migration assets never re-bought): keep the current static fallback path.
- UI consuming this (`DividendStats` / dividends page): closed positions render with a
  `Chiusa` outline chip; keep the list ranked as today.

## 7. Shared accounts & demo

- Everything is owner-scoped: `ownerId` from `useActiveAccount()` everywhere; a delegate CAN
  register trades (route uses `assertCanAccessAccount`). Manual `useEffect` loaders (if any are
  added) must include `ownerId` in deps (the switching gotcha).
- Demo user: sees the migrated ledger and the movements list; every mutation control disabled via
  `useDemoMode()` with the standard aria-label.
